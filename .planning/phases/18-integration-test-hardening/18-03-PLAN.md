# Plan 18-03: Adversarial & Negative Input Stress Tests

## Goal

Stress-test every CLI tool's error paths with user-mistake scenarios: malformed inputs, schema mismatches, SSSOM audit-log mistakes, CLI misuse, unit-detection pitfalls, YARRRML fixture hygiene. Every adversarial test asserts exit code 1 (or explicit 0 for silently-recovered cases), a diagnostic message in `stderr`, and **no partial output file written** on failure. Includes six mocked `rosetta-translate` error-path tests.

## Must-haves

### Truths

1. A developer can run `uv run pytest rosetta/tests/adversarial/` and see ≥17 green tests across 6 categories: malformed inputs (~4), schema mismatches (2) [review — missing_required removed, doesn't exist in lint], SSSOM mistakes (~4), CLI misuse (~4), unit pitfalls (~4), translate errors (6).
2. Every malformed-input test asserts `result.exit_code == 1`, a specific phrase in `result.stderr`, and **no output file written** (or if written, contains no partial artifact — checked by asserting file doesn't exist or is empty).
3. Ingesting a JSON file with a syntax error produces a stderr message naming the problem (line number or parser error), not an opaque `KeyError`.
4. Ingesting a truncated XML file produces a stderr message mentioning the XML parse failure.
5. A schema-mismatch suggest run (source schema claims `speed_kts: integer` but master expects `float`) produces a SUCCESS exit (0) because type-coercion warnings are downstream, but the following `rosetta-lint` run flags the `datatype_mismatch` finding in its JSON report.
6. `rosetta-accredit ingest` on a SSSOM TSV with duplicate MMC pairs for the same `(subject_id, object_id)` exits 1 with stderr naming the duplicate.
7. `rosetta-accredit ingest` on a SSSOM TSV with wrong column count exits 1 with a clear diagnostic.
8. `rosetta-accredit ingest` with a derank row (`owl:differentFrom`) referencing a `(subject_id, object_id)` pair that doesn't exist in the log exits 1 with a "cannot derank non-existent mapping" diagnostic.
9. CLI misuse tests: running `rosetta-yarrrml-gen --run` without `--data` exits 1 with "requires --data"; `rosetta-ingest` with `--input nonexistent.csv` exits 2 (Click's file-not-exists) not 1.
10. Unit-detection pitfall tests: a slot labelled `"signal_dbm"` is detected by `detect_unit` as a known unit but returns `None` for the IRI (NLP fallback case); a slot labelled `"height_metre"` (British spelling) is detected via NLP; an ambiguous slot `"count"` returns `None`.
11. YARRRML fixture-hygiene test: running `yarrrml-gen --run` with a master schema containing `range: dateTime` (the Phase 16-03 fixture typo) exits 1 with a diagnostic identifying the offending slot.
12. Six mocked `rosetta-translate` tests cover auth failure, quota exceeded, transient DeepL error, missing key + non-EN, EN passthrough without key (exit 0), empty schema (translator never called).
13. [review] Every adversarial test whose exit path involves a specific exception names the exception type in its docstring (e.g. "raises `deepl.exceptions.AuthorizationException`"). This enumeration is captured in each task's "Exception enumeration" block.
14. [review] No test asserts a rule code that doesn't exist in `rosetta/core/lint.py` or `rosetta/cli/lint.py`. Rule codes referenced: `datatype_mismatch`, `unit_dimension_mismatch`. `missing_required` and `unit_incompatible` are NOT asserted anywhere (they don't exist).
15. [review] Every adversarial test assertion on stderr uses stable project-controlled or library-API-name substrings (e.g. "authentication", "quota"), not CPython repr strings (like "Expecting") that drift across Python versions.
16. [review] `test_ingest_csv_with_bom_inline` asserts exit 0 with BOM stripped, consistent with 18-02 Truth #4. No ambiguous "verify current behavior" assertions remain.
17. [review] Preconditions (grep current rule codes, grep `owl:differentFrom`, grep translate stderr phrases) are explicitly listed at the top of each task's test block. Writing a test that fails these preconditions is a hard stop.

### Task 2 — `test_malformed_inputs.py`

   - No partial output

3. **`test_ingest_wrong_encoding_csv`** — `--format csv` on `wrong_encoding.csv`:
   - Exit 1, stderr mentions encoding/decode
   - No partial output

4. **`test_ingest_csv_with_bom_inline`** — inline build a CSV with a BOM using `tmp_path.write_bytes(b'\\ufeff...')`; ingest it. 5. **`test_suggest_empty_sssom_master`** — inline-create a LinkML master schema with zero classes; `rosetta-suggest` against it → either exits 0 with empty TSV or exits 1 with "no target classes" — whichever is current behavior.

**Done when:** All pass. Stderr assertions pin the exact diagnostic so a regression (e.g. exception message changes) is caught.

### Task 3 — `test_schema_mismatch.py`

**File:** `rosetta/tests/adversarial/test_schema_mismatch.py`

Tests (~3). **Precondition:** before writing, grep `rosetta/cli/lint.py` and `rosetta/core/lint.py` for the rule-code string table. Currently known codes: `datatype_mismatch` (lint.py:212), `unit_dimension_mismatch` (lint.py:171). Do not assert any other rule code until confirmed present.

1. **`test_suggest_type_divergence_flagged_by_lint`** — inline-create a source schema where `speed_kts: integer` but master has `speed_kts: float`; run `suggest` + `lint`. Assert:
   - `suggest.result.exit_code == 0`
   - `lint` JSON report contains at least one finding where `rule == "datatype_mismatch"` (confirmed code)
   - `lint` exit code: assert whatever the current severity-to-exit-code mapping produces — if `--strict=false` default yields exit 0 with WARNING findings, assert that; otherwise exit 1. Pin the observed value so a future severity-logic change breaks the test intentionally.

2. **`test_renamed_field_survives_as_alias`** — inline source schema with a slot `altitude_ft`; master has slot `altitude`. Suggest on the two:
   - `suggest.result.exit_code == 0`
   - SSSOM TSV produced; the row uses the aliased label correctly and `suggest` produces the expected predicate (`skos:broadMatch` or similar — confirm exact predicate from current suggest output before asserting).

**Note on the removed `missing_required` test:** the original plan proposed a third test asserting a `missing_required` lint finding. That rule code does not exist in `rosetta/core/lint.py` (grep-confirmed zero hits). The `required` constraint is a LinkML validation concern handled by `rosetta-validate` + SHACL shapes, not by `rosetta-lint`. Testing that path belongs in `test_validate_pipeline.py` (already in 18-02 Task 3 item 4) and is not duplicated here.

**Done when:** All pass. Each test names the expected severity/finding code exactly.

## Risks & mitigations

- **Risk:** Error messages change as dependencies update (e.g., `schema-automator` error text changes across versions). **Mitigation:** Assert on stable substrings, not full strings. Prefer "authentication" over "DeepL authentication failed: invalid auth key" — the former survives a dependency upgrade; the latter doesn't.
- **Risk:** Some "current behavior" cases (e.g. stdout/file collision in `yarrrml-gen`) may have unintended behavior we don't want to freeze. **Mitigation:** For each such test, write the docstring to make the observed behavior explicit ("As of Phase 18, this currently exits 0 with both streams merged — if this is wrong, update the CLI and then this test."). This makes intentional changes straightforward.
- **Risk:** The `unit_pitfalls` tests depend on Phase 17's `detect_unit` returning IRIs. **Mitigation:** Phase 17 is complete; confirm API shape before writing tests by running `uv run python -c "from rosetta.core.unit_detect import detect_unit; print(detect_unit('height_m'))"`.
